# 业务需求交付闭环工作流

当你需要承接一个完整的业务需求（从用户给需求链接/文档 → 实现上线 → 结项沉淀）时，使用本工作流。

## 前置条件

- Hermes tools: `terminal`, `file`, `web`, `search`, `git` 相关工具
- 项目记忆仓库: 每个需求开 feature 分支，暂存 requirements/plan/progress/验收记录
- 长期知识仓库: master 分支，存稳定业务知识（类比 memory + skill）
- GitHub PR/CR 工具链可用
- 测试框架已配置（pytest/go test/jest 等）

## 工作流概览：八阶段闭环

```
需求进入 → 澄清(第一质量门) → 技术方案(第二质量门) → TDD实现 → 内部质量门 → 验收 → 发布观察 → 结项沉淀
                                                                              ↑
                                                                          (反馈回流)
```

> 核心原则：每个阶段产出都基于上一阶段的产物继续推进。澄清不过不进方案，方案不过不写代码，验收不通过不发布。

---

## Step 1: 需求进入 — 先组织上下文

在开始任何工作之前，先获取两类信息：

### 1.1 当前需求材料
- 需求文档/链接/PRD → 用 `web_extract` 或 `browser_navigate` 获取内容
- 相关 issue/对话 → 从项目管理系统拉取
- 用户补充说明 → 直接对话中获取

### 1.2 长期业务知识
- 搜索 memory 中的相关历史记录（`session_search`）
- 搜索已有 skill 中是否有相关业务流程
- 读取项目知识库中的 wiki/codemap（如果有）

### 1.3 输出
把合并后的上下文整理成一份初始快照，包含：
- 需求要解决什么问题
- 相关历史/背景
- 系统边界和约束

> 不需要人工确认，除非事实源缺失或需求材料冲突。

---

## Step 2: 需求澄清 — 第一质量门

目标：把模糊需求变成可审阅的 requirements。

### 2.1 产出 requirements 文档
基于步骤 1 的上下文，生成结构化 requirements：

```markdown
## Requirements
- **目标行为**: 系统应该做什么
- **非目标边界**: 明确不做什么
- **验收标准**: 怎样算完成（每个标准要可验证）
- **风险**: 有哪些不确定之处
- **待确认问题**: 需要人工澄清的问题清单
```

### 2.2 协作留痕
- 把 requirements 提交到项目仓库的 feature 分支
- 人的反馈应该落到代码平台的评论体系（GitHub PR comment / CR 评论），不留在对话里

### 2.3 质量门
- 在 requirements 确认前：**不要进入技术方案或代码**
- 确认后的 requirements 是后续 plan、实现和验收的共同基准

---

## Step 3: 技术方案 — 第二质量门

目标：确认"怎么实现"，这是人工介入最密集的阶段。

### 3.1 产出 plan 文档

```markdown
## Technical Plan
- **现状**: 当前系统状态
- **目标行为**: 修改后的行为
- **不做什么**: 明确 scope
- **影响范围**: 涉及哪些模块/服务
- **核心改动**: 主要代码改动点
- **风险**: 技术风险 + 缓解措施
- **回滚策略**: 出问题怎么回滚
- **验收方式**: 如何验证（单元测试 + 接口 + 日志 + 监控）
- **配置依赖**: 开关/白名单/人群分流/配置 schema（必须在方案阶段确认！）
```

### 3.2 配置类需求的特别处理
- 读取当前配置状态（天机星/工作台等配置平台）
- 提前创建好配置的 schema
- 完成配置校验
- 把配置信息写进 plan 文档

### 3.3 测试策略必须在方案阶段定好
- 正面 case + 反面 case 都要写出
- 验收方式明确：单元测试、HSF 接口调用、SLS 日志查询、监控指标核对
- 验证入口是什么

### 3.4 质量门
- plan 走 CR 确认
- 确认前不进入编码
- 确认后的 plan 是实现、测试、验收的共同基准

---

## Step 4: TDD 实现

目标：测试驱动，而不是测试补全。

### 4.1 先写测试
- 根据 plan 里的验收标准写测试
- 测试定义"什么是对的"
- 测试应该是**业务行为约束**，不是覆盖率装饰

### 4.2 再写实现
- 让实现去满足测试
- 只改 plan 范围内指定的代码
- 频繁运行测试确认进度

### 4.3 提交策略
- 提交粒度小、可理解
- 每个 commit 有清晰的 message
- 只在本地验证通过后推送到远程

---

## Step 5: 内部质量门（自动，无需人工）

代码推送前，Agent 自己必须过这三关：

### 5.1 diff-to-test 映射
每处生产代码行为变化都要有具体的测试锚点：
- 新增函数？→ 单元测试覆盖
- 配置变更？→ 配置校验测试
- 分支逻辑？→ 条件组合测试

### 5.2 PMD / Lint 检查
```bash
# 运行项目的 lint/static analysis
npm run lint   # 或 go vet, pylint, cargo check 等
# 失败则回到实现阶段修复
```

### 5.3 Agent 内部代码 Review
PMD 通过后，对完整待 push diff 做结构化 review：
- 代码是否符合 plan？
- 是否有遗漏的边缘 case？
- 测试是否真的覆盖了行为？
- 发现 actionable 问题 → 修复 → 重跑测试 → 再 review

> 这些门禁不能只靠提示词约束，必须变成可执行的检查。

---

## Step 6: 验收

目标：证明"做对了"。

### 6.1 验收清单（基于 plan 中的验收方式）
- 单元测试全部通过
- HSF 接口调用正常（如适用）
- SLS 日志查询确认行为正确
- 监控指标核对（确认符合预期）
- 配置状态确认（已生效）

### 6.2 验收证据记录
- 把每项验收的截图/日志/输出记录到验收文档
- 记录到项目记忆仓库的验收目录

### 6.3 人工确认
- 让人确认验收证据
- 如有问题 → 回到相关阶段修复

---

## Step 7: 发布观察

### 7.1 灰度发布
- 先小范围观察

### 7.2 监控确认
- 检查业务监控指标
- 检查错误日志
- 确认无异常告警

### 7.3 自恢复
- 如果出现预期内的问题，Agent 尝试自恢复
- 如果出现意外问题，回滚并记录

---

## Step 8: 结项沉淀 — 反馈学习层

目标：让这次的教训变成下次的能力。

### 8.1 分析积累的材料
阅读整个需求过程中积累的：
- CR 评论（哪些被反复指出？）
- issue 对话
- 验收记录和人工修正
- 自恢复日志

### 8.2 三类输出

| 类型 | 去向 | 频率 |
|------|------|------|
| **稳定业务知识**（平台能力、业务口径、系统结构） | 蒸馏进长期 wiki / codemap / memory | 每次结项 |
| **流程/模式改进**（反复出现的问题、skill 缺口） | 更新或创建 skill | 每次结项 |
| **一次性材料**（本次需求的中间产物） | 归档到项目记忆仓库 | 保留备查 |

### 8.3 蒸馏进长期知识
把以下内容用 `memory(tool="add")` 或更新 skill 沉淀：
- 本需求涉及的**业务规则/口径**（memory）
- 本需求反复遇到的**流程坑**（skill patch）
- 本需求用到的**新模式/提示词改进**（skill update）

### 8.4 存档
- 项目记忆仓库的 feature 分支保留作为历史记录
- 删除不再需要的临时文件

---

## 关键原则

1. **不直接升级**：过程材料先留在项目记忆层（feature 分支），只有经过结项审视+人工确认，才选择性地进入长期知识（master）
2. **质量门的顺序不能跳过**：澄清→方案→实现→验收，每一步都是下一道的入场券
3. **反馈要落到平台评论体系**，不留在对话里 — 方便结项时统一回收分析
4. **配置在方案阶段处理**，不要在验收时才发现配置不对
5. **TDD 是质量约束，不是覆盖率装饰**：测试先于代码，测试定义正确行为

## 注意事项

- 本工作流适用于**中等复杂度以上的业务需求**。简单 bugfix（改一行配置、修一个 null 指针）不需要走完整流程
- 对于紧急线上问题，跳过 Step 1-3，直接修复 + 事后补沉淀
- 如果需求材料特别模糊，Step 2（澄清）可能需要多轮交互
- 结项沉淀是**最容易被跳过但最有长期价值的步骤**，务必执行
- skill 创建后需要定期审视（类似结项），避免 skill 过时或产生垃圾积累
